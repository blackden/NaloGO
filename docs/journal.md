# Журнал решений

Rolling-журнал того, **что и почему** мы выбираем при работе над форком `blackden/NaloGO`.
Не путать с `CHANGELOG.md` (там — что *изменилось* для пользователя). Здесь — *почему* мы это сделали.

**Формат записи:**

```
## YYYY-MM-DD — короткое название решения

**Контекст:** что мы делали, какие были варианты.
**Решение:** что выбрали.
**Почему:** мотивация — техническая, организационная или историческая.
**Последствия:** что теперь обязаны делать (или не делать) из-за этого выбора.
```

Если запись становится критически важной для будущих сессий — дублирую в `CLAUDE.md`. Журнал — широкий, `CLAUDE.md` — узкий «что обязательно надо знать».

## 2026-06-06 — Форк, а не «свой клиент с нуля»

**Контекст:** изначально собирались писать Python-обёртку над `lknpd.nalog.ru` с нуля, потому что в `~/fedorov_tech/moy-nalog/` был форк PHP-библиотеки `shoman4eg/moy-nalog`, а PHP — не «свой» язык. Перед стартом проверили, нет ли готовых решений.

**Решение:** взять готовую `nalogo` от `Rusik636/NaloGO` — это **полный порт того самого PHP-клиента** на тот же стек, который я бы и выбрал собирать сам (Python 3.11+ / httpx / pydantic v2 / asyncio / mypy / ruff / pytest+respx / bandit / CI). Делаем форк `blackden/NaloGO`, начинаем работать оттуда.

**Почему:** переоткрытие велосипеда. У `nalogo` 1.0.0 уже есть обе схемы авторизации, decimal-арифметика для денег, 88% покрытие, безопасное логирование с маскировкой токенов, CI-pipeline. Если делать с нуля — потратим недели, получим то же самое.

**Последствия:**
- Семантический референс остаётся **PHP-библиотека** `shoman4eg/moy-nalog`: в каждом докстринге написано «Maps to PHP X::y()». Когда непонятно «что API ожидает» — открываем PHP-исходник, а не гадаем по Python-порту.
- Подключаем `upstream` remote на `Rusik636/NaloGO`, чтобы тянуть фиксы автора без ручных PR-мержей.
- Сам PHP-форк (старый репо `moy-nalog`) в архив.

## 2026-06-06 — Стек: `uv` + ruff + black + mypy + pytest + respx + bandit

**Контекст:** надо зафиксировать стандартный набор инструментов разработки, чтобы потом не было «а почему ты не запускал X».

**Решение:** придерживаемся того, что уже в `pyproject.toml` апстрима, плюс `uv` для управления окружением вместо голого `pip`.

| Инструмент | Зачем |
|---|---|
| `uv` (Astral) | Виртуальное окружение, установка пакетов, в перспективе — `uv.lock`. Глобальный стандарт ragnar'a (см. `~/.claude/CLAUDE.md`). Заменяет `pip` / `pipx` / `venv` / `poetry`. |
| `ruff` | Линтер. На 3.11+ автоматически активирует `UP*` правила (модернизация). |
| `black` | Форматтер. Из-за `--check` в CI — обязателен. |
| `mypy` | Строгая статическая типизация (`disallow_untyped_defs = true`). |
| `pytest` + `pytest-asyncio` (`asyncio_mode=auto`) | Тесты. Без `@pytest.mark.asyncio` на каждом тесте. |
| `respx` | Мок HTTP-слоя поверх `httpx`. Не `pytest-httpx`. |
| `bandit` | Сканирование безопасности. В CI есть и обязательно. |

**Почему:** ровно то, что уже принято апстримом и работает. Менять без причины — генерировать конфликты при `git fetch upstream`.

**Последствия:**
- Любая новая зависимость пополняет `pyproject.toml`, ставится через `uv pip install -e ".[dev]"`.
- При смене `python_requires` (сейчас `>=3.11`) — обновлять `[tool.mypy] python_version`, `[tool.ruff] target-version`, `[tool.black] target-version` синхронно. Это три места.

## 2026-06-06 — Baseline зелёного билда зафиксирован, кроме `ruff UP042`

**Контекст:** прогнал все CI-шлюзы локально, чтобы понимать, что значит «зелёное», прежде чем что-то трогать.

**Решение:** зафиксировать baseline:
- `pytest`: 46/46
- `black --check`: чисто
- `mypy nalogo/`: чисто
- `bandit -r nalogo/ -ll`: 0 issues
- `coverage`: 80.74% (порог 80% впритык)
- **`ruff check .`: 3 ошибки `UP042`** — `class X(str, Enum)` в `nalogo/dto/income.py` (l.15 `IncomeType`, l.22 `PaymentType`, l.29 `CancelCommentType`)

При этом CI у апстрима **зелёный** на этих же файлах.

**Почему расхождение:** в `.github/workflows/ci.yml` стоит `pip install ruff black` **без пиннинга версий**. У апстрима CI бежит с ruff, в котором `UP042` либо отсутствовал, либо был не в дефолтном наборе. Локально у нас `ruff 0.15.16` — флагает.

**Последствия:**
- Зелёный CI у апстрима — **«гнилое зелёное»**: результат зависит от версии инструмента на момент конкретного запуска. Сегодня прошёл — завтра, когда GitHub Actions обновит образ или ruff выкатит новую версию, упадёт без единого коммита.
- Отдельная задача в очередь: **закрепить версии `ruff`, `black`, `mypy`, `bandit` в `pyproject.toml`** (через `dev`-секцию с минимальными версиями → перейти к `>=X.Y.Z,<X+1.0.0`), либо генерировать `uv.lock`, либо оба варианта.
- Три `UP042` — отдельная микрозадача: заменить `class X(str, Enum)` на `class X(StrEnum)`. **Перед этим** надо понять, не сломает ли это сериализацию в `model_dump()`/`json` (поведение `StrEnum.__str__` отличается от `str(value)` у `class X(str, Enum)` — нужен явный pydantic-тест). Сейчас не трогаем, потому что baseline и так зелёный в апстримном CI; зафиксировали как известную ловушку.
- Coverage 80.74% — **впритык к гейту 80%**. Любое добавление кода без тестов может уронить под порог. Когда трогаем `nalogo/payment_type.py` (43% покрытия) или `nalogo/tax.py` (44%) — добавляем тесты в том же PR.

## 2026-06-06 — Известные ловушки апстрима, которые НЕ чиним сразу

Зафиксировано здесь, чтобы не «забыть и переоткрыть» через месяц:

1. **`.coverage` закоммичен в репо**, хотя в `.gitignore`. `.gitignore` не отслеживает уже-tracked файлы. Чистка: `git rm --cached .coverage` в отдельном коммите.
2. **`.gitignore` блокирует `*.json` целиком** с белым списком. Любая JSON-фикстура потребует явного `!path/...`. Не лечим — это сознательная защита от утечки `tokens.json`.
3. **`pyproject.toml` `[project.urls]` и `authors` — плейсхолдеры** (`your-org/nalogo`, `contributors@nalogo.com`). Чинить **только** перед публикацией нашего форка на PyPI под собственным именем.
4. **CI триггерится на легаси-ветках `migrate/python-async*`** — миграционный мусор. Снести при следующей правке `ci.yml`.
5. **`httpx.AsyncClient()` создаётся на каждый запрос** (нет переиспользования пула). Для текущей нагрузки API это не больно; оптимизировать только если упрёмся.
6. **`refresh()` глотает все исключения** (`except Exception: return None`) — намеренно, под PHP-поведение. Менять — отдельное обоснованное решение.
